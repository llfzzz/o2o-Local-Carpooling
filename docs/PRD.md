# PRD

## 目标

构建企业级 O2O 同城拼车 MVP，形成「手机号登录 -> 司机证件审核 -> 车主发布行程 -> 乘客订座 -> 模拟支付 -> 超时取消/库存释放 -> 后台复核审计」闭环。

## 角色

- 乘客：登录、搜索路线、查看行程、锁座、模拟支付、取消订单、查看我的行程。
- 司机：提交资质、发布路线、管理座位、确认订单、取消发车。
- 运营：司机审核、订单监控、行程管理、用户管理、文件/OCR 复核、审计追踪。
- 系统：执行超时取消、事件投递、状态机校验、幂等保护、日志追踪。

## MVP 范围

- 手机号验证码登录，验证码与 JWT 首期为 Mock/占位实现。
- RBAC 角色模型：RIDER、DRIVER、OPERATOR、ADMIN。
- 司机驾驶证/行驶证上传与 OCR Mock。
- 后台人工通过/驳回司机认证。
- 车主发布行程，保存路线快照、距离、时长、价格和库存。
- 乘客按起终点和时间搜索行程。
- 订单创建时锁座，模拟支付成功后确认订单。
- 支付超时释放库存；当前 `0.3.0` 由 order-service 定时扫描兜底，RabbitMQ 延迟取消作为后续演进设计。
- MinIO 私有桶文件存储设计。
- MongoDB 审计日志设计。

## 非目标

- 首期不接真实支付。
- 首期不接真实实名、活体、人脸核验。
- 首期不做复杂动态拼单调度。
- 首期不做原生 App。

## 准确性

- 路线距离、价格、座位库存和订单状态以服务端为准。
- 地图供应商结果必须保存 `RouteSnapshot`，不能只依赖前端临时数据。
- OCR 结果必须保存供应商、置信度、字段快照和原文件引用。
- 订单状态只能通过状态机迁移，禁止接口直接覆盖状态字段。

## 安全

- 所有外部流量走 Gateway `/api/**`。
- JWT/RBAC 在 Gateway 和业务服务双层校验。
- 创建订单、支付回调、审核动作需要幂等键。
- 文件对象默认私有，只返回短时授权 URL 或文件 ID。
- 手机号、车牌、证件号加密存储，列表默认脱敏。
- 后台操作写审计日志，关键事件带 traceId。
